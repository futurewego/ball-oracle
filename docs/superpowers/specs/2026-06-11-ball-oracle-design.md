# ball-oracle 设计方案（球神谕 / AI 神算子）

> 日期：2026-06-11
> 状态：待用户评审
> 背景：2026 美加墨世界杯已于 2026-06-11 开赛，热度窗口约 3-4 周。

---

## 1. 目标与定位（Why）

做一个**可对外的轻产品 / MVP**，用「数据 + LLM 混合」的方式做世界杯预测，验证「AI 预测 + 社交」的业务假设，并借世界杯热度抢传播窗口。

核心约束：**热度窗口仅 3-4 周，能否上线 > 传播天花板 > 架构完美度。**

## 2. 范围与分波（Scope）

用户期望覆盖 4 块内容，但全做会样样半成品。按「先 MVP 再优化」分两波：

**第一波（抢热度，唯一交付目标）= 产品核心闭环**
- 数据层：赛程 + 实时赔率 + 球队战绩
- 预测引擎：赔率反推概率 + ELO 打底 → LLM 生成人话分析
- 赛前预测卡：每场一张可分享图卡（传播钩子）
- 用户预测 + PK 排行榜（留存钩子）

**第二波（热度起来后）**
- 夺冠 / 出线概率沙盘（赛事级宏观，数据更新慢，可后置）
- 赛后 AI 自动复盘（依赖第一波积累的预测记录）

理由：单场预测 + 排行是「每天有新内容」的高频钩子；夺冠概率、复盘是低频/衍生内容，技术上也依赖前者先跑起来。

## 3. 产品形态（How - 交付）

**先 Web 后补小程序。** Web App 最快上线、分享即传播、无审核。架构上把预测引擎与业务逻辑做成独立 API 层，前端仅为壳，后续包小程序时后端直接复用。

## 4. 架构总览

```
┌─────────────┐     ┌──────────────────┐     ┌─────────────┐
│ Web 前端     │────▶│  API 层 (复用核心) │────▶│ Supabase    │
│ Next.js     │     │  - 预测引擎        │     │ Postgres    │
│ (后续小程序  │◀────│  - 用户/排行       │◀────│ Auth/Realtime│
│  复用此API)  │     │  - 内容生成        │     └─────────────┘
└─────────────┘     └────────┬─────────┘
                             │
                   ┌─────────┴─────────┐
                   ▼                   ▼
            [体育数据API]         [Claude LLM]
            赛程/赔率/战绩         分析+复盘生成
```

**关键原则**：预测引擎是独立模块（输入数据 → 输出概率 + 分析的纯逻辑），不绑前端、可单测、可被任何端调用。

## 5. 模块设计

### 5.1 数据层
- **赛程 + 战绩**：football-data.org（含世界杯赛事，免费档）
- **实时赔率**：The Odds API（赔率是混合方案「数据基线」的核心）
- **缓存策略**：赛程/球队数据低频拉取存 Supabase；赔率赛前定时刷新。不每次请求都打第三方（省额度 + 稳）
- ⚠️ **待办**：各 API 免费档具体配额需先 WebSearch 实测确认后再锁定选型，不拍脑袋。

接口契约：`数据层` 对外暴露 `getFixtures()`、`getOdds(matchId)`、`getTeamStats(teamId)`，返回标准化结构，屏蔽底层数据源差异（便于换源）。

### 5.2 预测引擎（核心护城河，三步）
1. **数据基线**：赔率反推隐含概率 + 球队 ELO 战力 → 客观胜率
2. **LLM 增强层**：把基线概率 + 近期战绩 + 伤停/对阵历史喂给 Claude，让它 ①给出人话锐评 ②指出基线没覆盖的因素 ③可微调概率并说明理由
3. **输出**：每场一个结构化预测对象

```
PredictionResult {
  matchId: string
  probabilities: { home: number, draw: number, away: number }  // 和为 1
  predictedScore: { home: number, away: number }
  commentary: string        // AI 人话锐评
  confidence: number        // 信心指数 0-1
  reasoning: string         // LLM 微调理由（可选展示）
}
```

接口契约：`predict(matchData) -> PredictionResult`，纯逻辑、可单测（基线层可不依赖 LLM 单独测）。

### 5.3 用户与社交层
- Supabase Auth 登录（邮箱 / 第三方）
- 用户对每场提交预测：`submitPrediction(userId, matchId, prediction)`
- **排行榜**：按命中率 / 积分排名，AI 作为「虚拟选手」上榜 → 制造「能不能赢过 AI」的钩子
- Realtime 推送比分与排名更新

积分规则（MVP 简化版）：胜平负命中 +1，比分全中 +3。结算在比赛结果落库后触发。

### 5.4 内容 / 传播层
- **AI 预测图卡**：每场生成可分享图（队徽 + 预测比分 + 胜率条 + 一句锐评 + 二维码），一键分享
- 图卡分析结果缓存，不重复调用 LLM

## 6. 数据流（单场预测生命周期）

1. 定时任务拉取赛程 → 存 Supabase
2. 赛前刷新赔率 → 触发预测引擎 → 生成 `PredictionResult` 存库
3. 生成预测图卡（缓存）
4. 用户浏览/提交自己的预测
5. 赛后拉取真实比分 → 结算用户与 AI 积分 → 更新排行榜（Realtime 推送）

## 7. 技术栈

| 层 | 选型 | 理由 |
|----|------|------|
| 前端 | Next.js (App Router) + Tailwind | 快、SSR 利于分享卡 SEO |
| 后端 | Next.js API Routes | 一体化、省运维；逻辑模块化便于抽离 |
| DB/Auth | Supabase | Postgres + Auth + Realtime 开箱即用 |
| LLM | Claude Sonnet（复盘可用 Opus） | 高频分析用 Sonnet 控成本 |
| 部署 | Vercel | 一键、零运维 |

## 8. 错误处理与可靠性

- 数据 API 失败：用最近一次缓存数据兜底，标记「数据可能延迟」
- LLM 失败：降级为仅展示数据基线概率（无锐评），不阻塞预测产出
- 赔率缺失的比赛：仅用 ELO 基线，信心指数自动调低

## 9. 测试策略

- **预测引擎**：纯逻辑单测，基线层（赔率反推 + ELO）独立于 LLM 测试，给定输入断言概率输出
- **积分结算**：单测各种命中场景（胜平负中/比分全中/全错）
- **数据层**：mock 第三方 API 响应，测标准化转换
- LLM 增强层：契约测试（结构合法、概率和为 1），不断言具体文案

## 10. 风险与成本

- **数据 API 配额**：最大不确定项，第一步先实测（见 5.1 待办）
- **LLM 成本**：每场分析约几千 token，Sonnet 可控；图卡分析缓存不重复生成
- **时间**：第一波闭环（数据 → 预测 → 图卡 → 排行）是唯一目标，其余一律后置
